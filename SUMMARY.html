Protocol Architecture Summary

1. The Big Picture

This project is basically a custom UDP protocol written in C++ that tries to get video frames across a bad network as fast as possible without losing them. The hardest part was keeping the playout delay low while making sure we don't accidentally send more than double the original bandwidth. I used a token bucket to carefully manage how much extra data I was sending so it stays right under that 2.0x limit.

2. Packet Layout

The normal frame from the harness is 164 bytes (4 bytes for the sequence number and 160 for the payload). Since I can only send about 320 bytes per frame, I can't just send two full packets every time. Instead, I combined them into a single custom packet to save space on headers:

  • Standard Packet (162 bytes): I shrank the sequence number down to 2 bytes. The highest bit of this number acts as a flag to say if there's a second payload attached. Then comes the 160-byte payload.
  • Packed Packet (324 bytes): If that flag is set to 1, the packet just tacks on another 2-byte sequence number and another 160-byte payload right at the end.

This setup lets me squeeze two frames into a single UDP packet without going over the strict budget.

3. How the Sender Works

The sender just keeps an array of all the old frames. To stay under the bandwidth limit, it uses a token bucket system where it gets 315 tokens per tick.

  • Sending Extra Data: Whenever it has at least 324 tokens saved up (which is almost all the time), it sends a Packed Packet.
  • Picking What to Send: First, it checks if the receiver sent a NACK asking for a missing frame. If there's a NACK, it sends that frame. If everything is fine, it just grabs the frame from 3 steps ago and sends that. This simple trick fixes most short dropouts without even needing to wait for a NACK.

4. How the Receiver Works

The receiver code ended up being super simple. It turns out the harness player doesn't care if you give it frames early, it just cares that they aren't late.

  • Passing Frames: Whenever a packet comes in, the receiver unpacks it, checks if it has seen those sequence numbers before, and if not, it instantly forwards them to the player. No messy timers or holding buffers needed.
  • Asking for Missing Frames: It remembers the highest sequence number it has received. Every 20ms or so, it checks if it skipped any numbers recently. If it sees a gap, it packs those missing numbers into a small packet and shoots it back to the sender as a NACK.
